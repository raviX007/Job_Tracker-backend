# Deployment

## Render (Primary)

The API is deployed on Render's free tier with a Neon PostgreSQL database.

### Configuration File: `render.yaml`

```yaml
services:
  - type: web
    name: job-agent-api
    runtime: python
    plan: free
    buildCommand: pip install -r requirements.txt
    startCommand: uvicorn api.server:app --host 0.0.0.0 --port $PORT
    healthCheckPath: /api/health
    envVars:
      - key: DATABASE_URL
        sync: false
      - key: API_SECRET_KEY
        generateValue: true
      - key: ALLOWED_ORIGINS
        sync: false
      - key: PYTHON_VERSION
        value: "3.12"
```

### Required Environment Variables (Render)

| Variable | How to Set | Notes |
|----------|-----------|-------|
| `DATABASE_URL` | Manual | Neon connection string with `?sslmode=require` |
| `API_SECRET_KEY` | Auto-generated | Or set manually |
| `ALLOWED_ORIGINS` | Manual | Your dashboard URL |
| `PIPELINE_SERVICE_URL` | Manual | Pipeline microservice URL |
| `LOG_FORMAT` | Manual | Set to `json` for structured JSON logs |
| `GMAIL_ADDRESS` | Manual | For email sending |
| `GMAIL_APP_PASSWORD` | Manual | Gmail App Password |
| `EMAIL_SENDING_ENABLED` | Manual | `true` or `false` |
| `RESUME_PATH` | Manual | Path to resume PDF on Render |
| `SENTRY_DSN` | Manual | Sentry DSN (empty = disabled) |
| `SENTRY_ENVIRONMENT` | Manual | Set to `production` |
| `SENTRY_TRACES_SAMPLE_RATE` | Manual | `0.1` recommended for production |

### Security Headers in Production

The API's `SecurityHeadersMiddleware` automatically adds HSTS (`Strict-Transport-Security`) when it detects HTTPS — which Render provides via its `x-forwarded-proto` header. No additional configuration needed.

### Health Check

Render pings `GET /api/health` to verify the service is running. This endpoint:
- Does NOT require authentication
- Checks database connectivity
- Returns pool stats, version, and uptime

```json
{
  "status": "ok",
  "db": true,
  "version": "1.0.0",
  "uptime_seconds": 3600,
  "pool": { "size": 2, "free": 1, "used": 1, "min": 2, "max": 10 }
}
```

### Cold Start

Render free tier spins down after 15 minutes of inactivity. First request after spin-down takes 30-50 seconds. The API client uses a 60-second timeout to handle this.

## Neon PostgreSQL

### Free Tier Limits

| Limit | Value |
|-------|-------|
| Storage | 0.5 GB |
| Compute hours | 191.9 hours/month |
| Branches | 10 |
| Auto-suspend | After 5 minutes of inactivity |

### Connection String Format

```
postgresql://user:password@ep-xxx-yyy.us-east-2.aws.neon.tech/job_tracker?sslmode=require
```

SSL is **required** — the API uses `ssl="require"` when creating the asyncpg pool.

### Connection Pooling

```python
pool = await asyncpg.create_pool(
    dsn=DATABASE_URL,
    min_size=2,                           # Keep 2 connections warm
    max_size=10,                          # Max 10 concurrent connections
    ssl="require",
    command_timeout=30,                   # Kill hung queries after 30s
    max_inactive_connection_lifetime=300, # Refresh stale connections every 5 min
)
```

Neon free tier supports up to 20 concurrent connections. The pool max of 10 leaves headroom.

- `command_timeout=30` prevents any single query from blocking a connection indefinitely.
- `max_inactive_connection_lifetime=300` ensures connections are refreshed before Neon's auto-suspend drops them.

## Local Development

### Prerequisites

| Requirement | Version |
|-------------|---------|
| Python | 3.12+ |
| PostgreSQL | 14+ (local) or Neon (remote) |

### Reproducible Deploys with Lockfile

A fully pinned `requirements.lock` is generated by `pip-compile` (from pip-tools). For reproducible production deploys, use the lockfile:

```bash
pip install -r requirements.lock
```

For development (allows newer patch versions), use the standard requirements:

```bash
pip install -r requirements.txt
```

To regenerate the lockfile after changing `requirements.txt`:

```bash
pip-compile requirements.txt -o requirements.lock --strip-extras
```

### Quick Start

```bash
cd api

# 1. Install dependencies
pip install -r requirements.txt

# 2. Configure
cp .env.example .env
# Edit .env — at minimum set DATABASE_URL

# 3. Start the server
uvicorn api.server:app --host 0.0.0.0 --port 8000 --reload
```

Run Alembic migrations after first setup:

```bash
# New database: apply schema.sql first, then stamp
alembic stamp head

# Existing database: run any pending migrations
alembic upgrade head
```

### Using Local PostgreSQL

```bash
# Create database
createdb job_tracker

# Connection string (no SSL needed locally)
DATABASE_URL=postgresql://anuraj@localhost/job_tracker

# Leave API_SECRET_KEY empty to disable auth for local dev
API_SECRET_KEY=
```

### Using Neon for Local Dev

Use the same Neon connection string as production, or create a Neon branch for development.

## Docker

A `Dockerfile` and `.dockerignore` are included for containerized deployment.

### Build

```bash
cd api
docker build -t job-tracker-api .
```

### Run

```bash
# Using .env file
docker run -d --name job-tracker-api --env-file .env -p 8000:8000 job-tracker-api

# Or pass environment variables individually
docker run -d --name job-tracker-api \
  -e DATABASE_URL="postgresql://user:pass@host/db?sslmode=require" \
  -e API_SECRET_KEY="your-secret-key" \
  -e ALLOWED_ORIGINS="http://localhost:8501" \
  -p 8000:8000 \
  job-tracker-api
```

### Verify

```bash
curl http://localhost:8000/api/health
# → {"status": "ok", "db": true, "version": "1.0.0", "uptime_seconds": 42, "pool": {"size": 2, ...}}
```

API docs: **http://localhost:8000/docs**

### Dockerfile

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "api.server:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Custom Port

To run on a different port, override the command:

```bash
docker run -d -p 9000:9000 job-tracker-api \
  uvicorn api.server:app --host 0.0.0.0 --port 9000
```

### Stop & Remove

```bash
docker stop job-tracker-api && docker rm job-tracker-api
```

## Monitoring

### Logs

The API uses structured logging via `core/logger.py` with two output modes controlled by `LOG_FORMAT` env var:

- **`LOG_FORMAT=console`** (default): Colored, human-readable output for local development
- **`LOG_FORMAT=json`**: Structured JSON output for production (Render, Datadog, etc.)

Set `LOG_FORMAT=json` on Render to get structured logs with `service`, `request_id`, and Datadog APM correlation fields (`dd.trace_id`, `dd.span_id`).

File handler always writes JSON regardless of `LOG_FORMAT`. See [`logging.md`](logging.md) for details.

### Key Log Events

| Event | Level | When |
|-------|-------|------|
| Pool created | INFO | Server startup |
| Tables verified | INFO | Server startup |
| Email sent | INFO | After SMTP success |
| Email failed | ERROR | After SMTP failure |
| API key rejected | WARNING | Invalid auth attempt |
| Rate limit exceeded | WARNING | Client hits 60 req/min |
| Unhandled exception | ERROR | Unexpected server error (logged with request_id) |

### Health Monitoring

- **Render:** Auto-pings `/api/health` every 30 seconds
- **Manual:** `curl https://your-api.onrender.com/api/health`
